Fase 1 · Análisis

Contexto y Problema

📋

¿Qué es esta práctica?

Nuestra clase actúa como el estudio de desarrollo EcoCode Studio. Un cliente ficticio quiere desarrollar una app de gestión de eventos (conciertos, bodas, conferencias, festivales, ferias y eventos corporativos), pero impone una condición clara:

El software debe diseñarse siguiendo los principios de la Economía Circular (7R). No basta con que funcione: debe ser sostenible, eficiente y responsable.

Esta presentación recoge nuestra propuesta como consultores de sostenibilidad digital.

❗

Problema que detectamos

  • Las apps de eventos actuales duplican funcionalidades sin optimizarlas
  • Altísimo consumo de servidores por consultas no optimizadas
  • Sistemas monolíticos imposibles de actualizar sin reescribirlos
  • Datos de eventos pasados abandonados (no reutilizados ni aprovechados)
  • Dependencia de plataformas propietarias costosas
  • Ciclos de vida cortos: apps sustituidas cada 2-3 años generando residuos digitales

Fase 1 · Nuestra propuesta

Objetivos de Nuestra Propuesta

🌱

Sostenibilidad Digital

Diseñar un sistema que minimice el consumo energético, optimice los recursos de servidor y tenga el menor impacto ambiental posible en todo su ciclo de vida.

⚙️

Eficiencia Técnica

Arquitectura modular y escalable que permita crecer sin reescribir el sistema. Código limpio, mantenible y documentado para reducir la deuda técnica.

♻️

Aplicación Real de 7R

Cada decisión de diseño arquitectónico debe justificarse bajo al menos una de las 7R. No se toman decisiones técnicas sin evaluación de impacto circular.

🤝

Colaboración Abierta

Priorizar herramientas open source, APIs abiertas y estándares públicos. Evitar dependencias propietarias que generen obsolescencia forzada.

📊

Argumentación Ambiental

Toda decisión técnica debe poder defenderse desde una perspectiva ambiental medible con métricas concretas de sostenibilidad.

🔁

Longevidad del Sistema

El sistema debe diseñarse para durar 10+ años mediante actualizaciones incrementales, sin necesidad de sustituciones completas.

Fase 1 · Nuestra propuesta de sistema

Descripción del Sistema

Nuestra propuesta es diseñar EventFlow como una Progressive Web App (PWA) con arquitectura de microservicios. No programamos la app entera, pero aquí describimos cómo la diseñaríamos para que sea sostenible desde el primer día.

Módulos Funcionales

🗓️
Gestión de Eventos

Creación, edición y publicación de eventos (conciertos, bodas, conferencias, festivales). Gestión de fechas, venues y capacidades.

🎟️
Venta de Entradas

Sistema de ticketing integrado con QR reutilizables, reducción de papel impreso y validación digital en puerta.

👥
Gestión de Participantes

Registro, listas de invitados, check-in digital, gestión de roles (asistente, proveedor, voluntario, artista).

📦
Proveedores y Recursos

Gestión de catering, logística, montaje, técnicos de sonido/luz. Reutilización de proveedores en eventos recurrentes.

📈
Analítica Sostenible

Dashboard de métricas: asistencia, huella de carbono estimada, % digital vs físico, eficiencia energética del evento.

Arquitectura Técnica

Frontend PWA Vue.js 3 + Vite · Service Worker · Offline-first
⬇ API REST + GraphQL
Backend Microservicios Node.js · Docker · API Gateway
⬇ ORM optimizado
Capa de Datos PostgreSQL · Redis Cache · S3 optimizado
⬇ CI/CD sostenible
Infraestructura Verde Servidores 100% renovable · Auto-scaling · CDN

Fase 1 · Análisis de impacto

Análisis de Impacto Ambiental

El software tiene impacto ambiental real. Cada petición al servidor, cada consulta de base de datos, cada archivo almacenado consume energía. EventFlow identifica y mitiga estos impactos desde el diseño.

⚠️ Impactos Potenciales

🖥️
Consumo de Servidores

Servidores activos 24/7 para eventos que ocurren puntualmente. Estimado: 2-8 kWh/día sin optimización.

📡
Tráfico de Red

Imágenes sin comprimir, vídeos de eventos, APIs sin cachear pueden generar cientos de MB por usuario/sesión.

🗄️
Almacenamiento Creciente

Datos de eventos pasados acumulados indefinidamente sin política de archivado o eliminación responsable.

📱
Obsolescencia Tecnológica

Apps que requieren actualizaciones constantes de hardware o SO fuerzan renovación de dispositivos innecesaria.

✅ Estrategias de Mitigación

⚡
Energía Renovable

Despliegue en proveedores cloud certificados con 100% energía renovable (Hetzner Green, OVH, AWS Green Regions).

💤
Auto-Scaling Inteligente

Escalar a cero fuera de periodos de alta demanda. Ahorro estimado del 70% en cómputo en horas valle.

🗜️
Compresión y CDN

WebP/AVIF para imágenes, Brotli para texto, CDN para reducir latencia y consumo de ancho de banda hasta un 60%.

📊
Dashboard de Huella

Medición en tiempo real del CO₂ estimado por evento. Objetivo: <0.5g CO₂ por visita de página (referencia GWF).

Fase 2 · Economía Circular

Las 7R Aplicadas

Para cada R proponemos decisiones técnicas concretas y justificadas. Haz clic en cada tarjeta para ver nuestra propuesta en detalle.

Decisiones Técnicas Concretas

  • Microservicios desacoplados: Módulo de ticketing, gestión de usuarios, notificaciones y analítica como servicios independientes. Si falla uno, no cae todo el sistema.
  • API-First Design: La lógica de negocio nunca está ligada al frontend. Cualquier cliente (web, móvil, IoT) consume la misma API.
  • Domain-Driven Design (DDD): El modelo de datos refleja el dominio real de eventos: Venue, Evento, Participante, Ticket, Proveedor, Recurso.
  • PWA con Offline-First: Funciona sin conexión para check-in en áreas con cobertura limitada (festivales de campo, bodas en masías remotas).
  • Diseño accesible (WCAG AA): Contraste, navegación por teclado y lectores de pantalla integrados desde el inicio, no añadidos después.

Impacto Ambiental

La arquitectura modular permite desactivar módulos no usados, reduciendo el footprint de cómputo activo. Estimamos una reducción del 35% en recursos de servidor respecto a una arquitectura monolítica equivalente.

Eficiencia arquitectónica
85%

Decisiones Técnicas Concretas

  • Optimización de consultas SQL: Índices estratégicos, paginación por cursor (no por offset), evitar N+1 queries con DataLoader pattern.
  • Lazy Loading agresivo: Imágenes, componentes JS y rutas cargadas solo cuando se necesitan. Reducción de bundle inicial: -60%.
  • Caché multicapa: Redis para sesiones y datos calientes, caché HTTP con ETags, Service Worker para assets estáticos.
  • Compresión de medios: Imágenes convertidas automáticamente a WebP/AVIF al subirse. Reducción media del 70% en peso.
  • Ticketing sin papel: QR digitales eliminan la impresión. Estimado: 0.5 toneladas CO₂/año evitadas por cada 100.000 entradas.
  • Notificaciones batch: Emails y SMS agrupados por lotes en lugar de enviarse uno a uno, reduciendo peticiones de API externas.

Impacto Ambiental

El ticketing digital y la eliminación de papel físico representa el mayor impacto. Para un evento de 10.000 personas: evitamos ~50kg de papel impreso y sus emisiones asociadas de transporte y residuos.

Reducción consumo red
70%
Reducción queries DB
55%

Decisiones Técnicas Concretas

  • Stack 100% open source: Vue.js, Node.js, PostgreSQL, Redis, Docker. Sin licencias propietarias ni vendor lock-in.
  • Design System compartido: Librería de componentes (botones, formularios, modales) reutilizada en web y en la app móvil.
  • Plantillas de eventos: Configuración de bodas, conciertos o conferencias guardada y reutilizable. Un organizador no rellena datos desde cero en cada evento.
  • APIs de pago consolidadas: Stripe/PayPal ya probados en lugar de implementar pasarela propia. Evitamos 6 meses de desarrollo y mantenimiento.
  • Autenticación OAuth2/OIDC: Reutilización de proveedores de identidad (Google, GitHub, email) sin base de datos propia de contraseñas.
  • Base de datos de venues: Integración con OpenStreetMap y datos públicos de espacios culturales municipales.

Impacto Ambiental

Reutilizar librerías consolidadas evita duplicar código que ya ha sido optimizado por miles de colaboradores. Estimamos que el 65% del sistema se apoya en código ya existente y probado.

% código reutilizado
65%

Decisiones Técnicas Concretas

  • Plan de mantenimiento trimestral: Sprint de refactorización dedicado cada 3 meses para eliminar deuda técnica acumulada.
  • Cobertura de tests ≥80%: Tests unitarios (Vitest), de integración y E2E (Playwright). Los bugs se detectan antes de llegar a producción.
  • Linting estricto: ESLint + Prettier obligatorios en CI/CD. Se rechazan merges con código de baja calidad.
  • Monitoring en tiempo real: Sentry para errores, Grafana para métricas de rendimiento. Los problemas se detectan y reparan en horas.
  • Documentación viva: OpenAPI/Swagger autogenerado desde el código. La documentación no queda desactualizada porque se genera automáticamente.
  • Política de zero-downtime deploys: Blue-Green deployments para actualizar sin interrumpir eventos en curso.

Impacto Ambiental

Cada bug corregido temprano evita costosas reescrituras futuras. Un sistema mantenido correctamente tiene una vida útil 3-4× mayor que uno sin mantenimiento, retrasando la generación de residuos tecnológicos.

Cobertura de tests
80%

Decisiones Técnicas Concretas

  • Versionado semántico (SemVer): Las APIs tienen versión explícita (v1, v2). Los clientes antiguos siguen funcionando mientras se migran gradualmente.
  • Feature flags: Nuevas funcionalidades activables/desactivables sin desplegar nuevo código. Rollout gradual al 1%, 10%, 100% de usuarios.
  • Dependencias actualizables: Dependabot automatiza alertas de seguridad y propone actualizaciones de dependencias con tests automáticos.
  • Migraciones de BD no destructivas: Solo se añaden columnas o tablas; nunca se eliminan en la misma versión. Rollback siempre posible.
  • Compatibilidad con dispositivos antiguos: Soporte para Android 8+ e iOS 13+. No forzamos actualización de hardware a los usuarios.

Impacto Ambiental

El soporte para dispositivos de hasta 6 años de antigüedad evita la obsolescencia forzada. Fabricar un smartphone genera entre 50-80kg de CO₂; alargar su uso útil es el mayor impacto positivo que puede hacer una app.

Cobertura de dispositivos
92%

Decisiones Técnicas Concretas

  • Importador de datos legados: Herramienta ETL para migrar datos de Eventbrite, Excel, o sistemas propietarios del cliente sin pérdida de información.
  • Historial de asistentes: Los datos de participantes de eventos pasados informan recomendaciones futuras (con consentimiento RGPD explícito).
  • Recuperación de plantillas: Configuraciones de eventos anteriores (catering, sonido, decoración) reutilizables automáticamente en eventos similares.
  • Analytics retroactivo: Los datos históricos permiten predecir aforo óptimo, reduciendo desperdicio de recursos (comida, sillas, personal).
  • API de integración bidireccional: Conexión con sistemas de gestión de espacios (Amadeus, Salesforce Events) ya en uso por el cliente.

Impacto Ambiental

La predicción de aforo con datos históricos reduce el desperdicio físico. En catering, un error de estimación del 10% en un evento de 500 personas genera ~50kg de comida desperdiciad. Los datos recuperados minimizan esto.

Datos históricos aprovechados
78%

Decisiones Técnicas Concretas

  • Publicación en npm/GitHub Packages: El sistema de ticketing, el módulo de notificaciones y el design system se publican como paquetes reutilizables.
  • Licencia MIT/Apache 2.0: Todo el código desarrollado expresamente para EventFlow se libera como open source, devolviendo valor a la comunidad.
  • Arquitectura plug-and-play: Los microservicios son contenedores Docker independientes que pueden "reciclarse" en otros proyectos (e-commerce, reservas hoteleras).
  • Documentación de patrones: Un libro de arquitectura interno documenta los patrones exitosos para que otros proyectos del estudio no partan de cero.
  • Política de datos al finalizar: Al cierre del servicio, los datos exportables se entregan al cliente. El código no se abandona, se archiva con documentación completa.

Impacto Ambiental

Publicar módulos como open source evita que otras empresas desarrollen lo mismo desde cero, ahorrando miles de horas de servidor en CI/CD y desarrollo redundante a nivel global.

Módulos reciclables

Fase 4 · Defensa

Conclusiones y Defensa

¿Por qué EventFlow es sostenible?

01

La sostenibilidad no es una característica añadida al final, es una restricción de diseño desde el día 0. Cada decisión arquitectónica tiene una justificación ambiental.

02

Eliminar el papel en eventos es el impacto más tangible: millones de entradas impresas sustituidas por QR digitales que funcionan offline.

03

La arquitectura modular no solo es técnicamente mejor: permite apagar lo que no se usa, reduciendo el consumo energético cuando la demanda es baja.

04

El código open source devuelve valor a la comunidad. Cada empresa que reutiliza nuestros módulos es CO₂ de desarrollo que no se genera.

Reflexión Final

"¿Puede el software ser sostenible aunque no sea un producto físico?"

Sí, y EventFlow es la prueba. El software consume energía real, genera datos que requieren almacenamiento físico y puede provocar o evitar la obsolescencia de hardware.

Un desarrollador responsable entiende que cada línea de código tiene impacto: en el servidor que la ejecuta, en el dispositivo que la muestra y en el planeta que genera esa electricidad.

La Economía Circular aplicada al software no es idealismo: es ingeniería de calidad con consciencia ambiental. Y EventFlow demuestra que ambas cosas son completamente compatibles.
